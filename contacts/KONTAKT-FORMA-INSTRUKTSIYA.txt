  ИНСТРУКЦИЯ: Рабочая отправка контактной формы (contacts.html)
  Защита от ботов: Yandex Smart Captcha (клиентский ключ в HTML, серверный — в
  config.local.php). Без обоих ключей отправка заявки невозможна. Почта: SMTP
  (PHPMailer), лимит по IP, honeypot. Сценарий ориентира: Beget (beget.com).
  Скрипт (submit.php) для Beget НЕ требует отдельной версии: достаточно
  config.local.php с реквизитами SMTP Beget, см. раздел 0.3.1.
  Актуальные порты и имена серверов проверяйте в базе знаний Beget.
  Дата-ориентир: внедрение по этапам, проверка после каждого блока.

--------------------------------------------------------------------------------
Установка почты для формы обратной связи (краткий чеклист)
--------------------------------------------------------------------------------

  Форма обратной связи на странице contacts/contacts.html отправляет заявки на
  сервер contacts/api/submit.php; письма уходят через SMTP (PHPMailer). Все
  настройки почты задаются только в contacts/api/config.local.php на хосте
  (в репозитории — образец contacts/api/config.local.sample.php).

  Минимум действий (подробности — раздел 0 и далее):

  1. Выложите на сервер папку contacts/ целиком вместе с корневым styles.css
     (см. шаг 0.1). Страница должна открываться по URL сайта (https://…), не
     через file://.

  2. На сервере скопируйте образец в рабочий конфиг:
       contacts/api/config.local.sample.php  →  contacts/api/config.local.php
     Файл config.local.php не коммитьте в git (он в .gitignore).

  3. Заполните в config.local.php параметры SMTP и получателей:
       smtp_host, smtp_port, smtp_secure ('ssl' | 'tls' | ''),
       smtp_user, smtp_pass,
       from_email (часто тот же ящик, что smtp_user), from_name,
       recipients — список e-mail, куда должны приходить заявки.
     Лимит антиспама: contact_max_per_window, contact_window_minutes.

  4. Укажите smartcaptcha_server_key (серверный ключ той же Yandex Smart
     Captcha, что и клиентский data-sitekey в contacts.html). Без него
     submit.php вернёт ошибку server_config, даже если почта заполнена.

  5. Папка contacts/api/data/ должна существовать и быть доступна PHP на запись
     (файл лимита запросов; см. шаг 0.4).

  6. Проверка: откройте contacts.html на сайте, пройдите капчу, отправьте форму;
     в ящиках из recipients должно появиться письмо (проверьте «Спам»).

  Ориентир для Beget: шаги 0.3.1–0.3.2. Внешний SMTP (не хостинг): раздел 2.3.


--------------------------------------------------------------------------------
0. Первая настройка скрипта (с нуля — пошагово)
--------------------------------------------------------------------------------

  Сделайте шаги по порядку; после каждого блока лучше проверить, что нет ошибок.

  Шаг 0.1. Файлы на хосте
    Убедитесь, что на сервере (в веб-корне сайта) лежит папка contacts/ целиком,
    в том числе:
      contacts/contacts.html
      contacts/contacts-form.js
      contacts/api/submit.php
      contacts/api/phpmailer/   (подкаталог src/ с библиотекой)
      contacts/api/config.local.sample.php
    И корневой styles.css (как в репозитории) — к нему подключается страница
    контактов. Без выкладки «живой» копии скрипты с компьютера письма не уйдут:
    открытие contacts.html через file:// не подойдёт.

  Шаг 0.2. Создать config.local.php
    На сервере (по SSH, FTP, файловому менеджеру) скопируйте файл
      contacts/api/config.local.sample.php
    в
      contacts/api/config.local.php
    Файл config.local.php НЕ кладите в git — он в .gitignore.

  Шаг 0.3. Заполнить настройки почты в config.local.php
    Общие поля (любой SMTP):
    • smtp_host, smtp_port — сервер и порт (см. 0.3.1 для Beget).
    • smtp_user, smtp_pass — логин (часто полный e-mail) и пароль ящика.
    • smtp_secure — 'ssl' для SMTPS (например порт 465), 'tls' для STARTTLS
      (часто 587 или 2525), '' — без шифрования (на проде лучше не использовать).
    • from_email, from_name — от кого письмо; from_email укажите ТОТ ЖЕ
      зарегистрированный на SMTP, с которого разрешён исходящий SMTP (у Beget —
      тот ящик, чей логин/пароль в smtp_user/smtp_pass).
    • recipients — все эти e-mail идут в обычное поле «Кому» (To); каждый
      видит остальные адреса в шапке письма (Bcc / скрытая копия не используется).
    • contact_max_per_window, contact_window_minutes — защита от флуда
      (по умолчанию 5 писем за 15 минут с одного IP).
    • smartcaptcha_server_key — серверный (секретный) ключ Yandex Smart Captcha
      (тот же экземпляр капчи, что и клиентский data-sitekey на странице).
      Без непустого значения submit.php не примет заявку (ошибка server_config).
      Создание капчи и выдача ключей: консоль Yandex Cloud, сервис Smart Captcha
      (документация: yandex.cloud → Smart Captcha).

  Шаг 0.3.0. Yandex Smart Captcha (кратко, подробно — раздел 9)
    В contacts/contacts.html у блока #contactSmartcaptcha задайте атрибут
    data-sitekey="…" (клиентский ключ, он публичный).
    В contacts/api/config.local.php добавьте 'smartcaptcha_server_key' => '…',
    взяв СЕРВЕРНЫЙ ключ в той же капче в облаке. Кнопка «Отправить» на странице
    остаётся неактивной, пока посетитель не пройдёт проверку; в JSON к API
    уходит поле captchaToken, на сервере — проверка через
    https://smartcaptcha.cloud.yandex.ru/validate

  Шаг 0.3.1. Почта и SMTP на Beget (beget.com)
    Код (submit.php, PHPMailer) менять не нужно: только правильно заполнить
    config.local.php на сервере.

    Актуальные хост, порты и варианты шифрования смотрите в базе знаний Beget,
    разделы про настройку почтовых клиентов / SMTP, например:
    https://beget.com/ru/kb/how-to/mail/
    (страница «Общие сведения» о серверах IMAP/SMTP, при необходимости — раздел
    с типичными портами и шифрованием.)

    Что сделать в панели Beget до заполнения config:
    1) Домен сайта прикреплён к аккаунту, почта на этом домене разрешена.
    2) Создать почтовый ящик, с которого сайт будет отправлять (например
       noreply@ваш-домен.tld) — «Почта» в панели → домен → «Почтовые ящики».
    3) Пароль ящика задать/скопировать и больше не светить (он пойдёт в
       smtp_pass на сервере, не в репозиторий).

    Типичные параметры для PHPMailer (проверьте в KB, если Beget обновит):
    • smtp_host: smtp.beget.com
    • Вариант A (часто рекомендуемый): smtp_port 465, smtp_secure 'ssl'
      (подключение SMTPS).
    • Вариант B (если 465 не подходит/блок у провайдера): smtp_port 2525
      (иногда 587), smtp_secure 'tls' (STARTTLS) — смотрите подсказку в KB/панели.
    • smtp_user: полный адрес ящика, напр. noreply@ваш-домен.tld
    • smtp_pass: пароль этого ящика
    • from_email: тот же полный адрес, что smtp_user (так Beget сопоставляет
      отправителя с авторизацией).
    • from_name: видимое имя, например «Сайт New Site».
    • recipients: один или несколько e-mail, куда слать заявки (сотрудники;
       не обязательно тот же ящик, что noreply — можно ящик приёма в другом
       поставщике, если внешняя доставка не запрещена; чаще всего — остаётесь
       внутри ящиков на том же домене Beget).

    Доставляемость: в панели Beget/почте для домена настройте SPF (и прочие
    записи, если панель предлагает) по инструкциям Beget, чтобы письма с формы
    не уходили в «Спам» у получателей. Это настройка DNS/почты, не PHP-файл.

  Шаг 0.3.2. Пример фрагмента config.local.php под Beget (замените домен/пароли/ключ)
    'smtp_host' => 'smtp.beget.com',
    'smtp_port' => 465,
    'smtp_user' => 'noreply@пример.рф',
    'smtp_pass' => '********',
    'smtp_secure' => 'ssl',
    'from_email' => 'noreply@пример.рф',
    'from_name' => 'Контакты сайта',
    'recipients' => [ 'inbox1@пример.рф', 'inbox2@пример.рф', ],
    'smartcaptcha_server_key' => 'серверный_ключ_Smart_Captcha_из_консоли_Yandex',

  Шаг 0.4. Права на каталог data
    Должна существовать папка:
      contacts/api/data/
    (в репозитории есть .gitkeep.) PHP должен уметь ЗАПИСЫВАТЬ в неё файл
    rate_state.json (создаётся автоматически). Обычно хватает прав 0755 на
    data/; если хостинг требует владельца веб-сервера — смотрите справку хостера.

  Шаг 0.5. Apache / Nginx
    Для Apache в contacts/api/ уже есть .htaccess: закрыт config.local.php и
    папка data. Если сервер Nginx, перенесите смысл правил (deny) в конфиг
    виртуального хоста, иначе убедитесь, что data/ и config не отдаются как
    статика нежелательным образом (часто PHP-only окружения это уже делают).

  Шаг 0.6. Проверка в браузере
    Откройте на сайте страницу:
      https://ваш-домен/.../contacts/contacts.html
    (как у вас настроен URL; главное — тот же домен, что и у размещённых файлов).
    Должны отображаться виджет Yandex Smart Captcha и кнопка «Отправить» в
    неактивном виде, пока проверка не пройдена. Заполните обязательные поля,
    пройдите капчу, затем отправьте форму. Успех: зелёное сообщение; иначе
    прочтите красный текст (коды captcha / server_config / send / rate — раздел 10).
    Проверьте почту всех addresses из recipients (и папку «Спам»).

  Шаг 0.7. Типовые ошибки
    «Сервер не настроен на отправку» (server_config) — нет config.local.php,
    пустой recipients, пустой smartcaptcha_server_key, или не хватает phpmailer
    в contacts/api/phpmailer/.
    Не настроен только клиентский ключ: подсказка в блоке капчи, кнопка
    «Отправить» не включится. Несоответствие пары data-sitekey / server key —
    ответ с кодом captcha.

2. Хостинг и почта (Beget; общий случай)
--------------------------------------------------------------------------------

  2.1. Сайт на Beget
       У проекта в этой инструкции ориентир — веб-корень на хостинге Beget, PHP
       включён (типично для shared и VPS у них). Скрипт submit.php — обычный
       PHP-файл, как и в catalog/api/ у каталога. Проверьте версию PHP в панели
       (желательно 8.0+). Исходящие соединения к smtp.beget.com должны быть
       разрешены (у Beget для SMTP это нормальная схема).

  2.2. Почта на Beget (рекомендуемый путь)
       Создаёте ящик на СВОЁМ домене в панели, используете smtp.beget.com и
       данные ящика в config.local.php — см. шаг 0.3.1. Отдельно «другой
       почтовый сервис» не обязателен: сайт и исходящая почта остаются в рамках
       одного аккаунта Beget.

  2.3. Почта НЕ на Beget (другой хостинг или внешний SMTP)
       Логика та же: в config вписываете smtp_*, from_email и smtp_user так, как
       требует ВАШ провайдер (Yandex 360, Mail.ru для бизнеса и т.д.). Скрипт
       не переписывается, меняется только config.local.php (и при необходимости
       правила файрвола исходящих на стороне хостинга).

  2.4. Список получателей (recipients)
       Несколько e-mail, на которые уходит заявка, задаются ТОЛЬКО в
       config.local.php; все перечисленные адреса отображаются в поле «Кому» (To)
       у каждого получателя. В JavaScript список не дублируется.
       Для внутренних ящиков Beget — создайте их в панели или укажите внешние
       адреса, если доставка разрешена.

  2.5. DNS и репутация
       Для Beget-ящиков и доменов в панели часто подсказаны/автодобавляются
       SPF и др. — следуйте разделу «Почта»/«Помощь» в KB. Вне Beget: SPF,
       DKIM, DMARC у регистратора или почтового вебхоста. Без корректных
       DNS-записей письма с формы чаще попадают в «Спам».

--------------------------------------------------------------------------------
5. Этап B: PHP-эндпоинт отправки
--------------------------------------------------------------------------------

  Шаг B.1. Создать contacts/api/submit.php, который:
           - Принимает только POST.
           - Читает тело как JSON: Content-Type: application/json, поля
             firstName, lastName, company, city, phone, email, comment,
             captchaToken (токен Yandex Smart Captcha),
             и опционально honeypot companyWebsite — пустая строка = ок.

  Шаг B.2. Проверка Content-Type/метода; при неверном методе — 405.

  Шаг B.3. json_decode, при ошибке — 400 + JSON { "ok": false, "error": "..." }
           (короткие сообщения, без внутренних деталей).

  Шаг B.4. Серверная валидация (зеркалируйте HTML5 и maxlength):
             firstName, company, phone, email, comment — обязательны;
             длины: как в форме (например 40, 1000 для comment);
             email — filter_var(FILTER_VALIDATE_EMAIL) или аналог;
             удаление/замена символов \r и \n в полях, которые могли бы попасть
             в заголовки (защита от header injection) — либо не использовать
             пользовательский ввод в Subject без строгой нормализации.

  Шаг B.5. Honeypot: если непустое скрытое поле (бот) — ответ 200 { ok: true }
           БЕЗ отправки письма (или 204), чтобы не подсказывать боту.

  Шаг B.5a. Yandex Smart Captcha: POST на https://smartcaptcha.cloud.yandex.ru/validate
           с параметрами secret (из config), token (из JSON), ip (REMOTE_ADDR);
           при status != "ok" — 400 { error: "captcha" }, письмо не отправлять.
           Выполнять после загрузки config и до rate limit, чтобы ложные попытки
           с неверной капчей не тратили лимит писем (счётчик rate — после проверки).

  Шаг B.6. Сборка письма:
             Тема: фиксированная с датой, например «Заявка с сайта New Site».
             Тело: plain text, все поля с подписями, кодировка UTF-8.
             Reply-To: e-mail из формы (после валидации).
             To: все адреса из recipients в поле «Кому»; скрытые копии нет.

  Шаг B.7. Отправка через PHPMailer + SMTP. При сбое SMTP — 502 или 500,
           в теле { ok: false }, в error — общая фраза; деталь логируйте
           в error_log на сервере, не в ответе клиенту.

  Шаг B.8. Успех: 200, Content-Type: application/json; charset=utf-8
           { "ok": true }.

  Шаг B.9. В начале файла: declare(strict_types=1); заголовок
           X-Content-Type-Options: nosniff (как в catalog/api/bootstrap.php).
           CORS: если запрос только с того же сайта, достаточно same-origin;
           если понадобится другой origin — тогда осторожно с Access-Control-*.


--------------------------------------------------------------------------------
6. Этап C: rate limiting
--------------------------------------------------------------------------------

  Шаг C.1. Создать contacts/api/data/ (на сервере — права на запись для PHP,
           например 0755 или как у catalog/api/data/).

  Шаг C.2. Хранить счётчики по IP: файл rate-limit.json или отдельные файлы,
           с блокировкой (flock) при записи, чтобы не портить при параллельных
           запросах.

  Шаг C.3. Правило: если IP превысил contact_max_per_window за окно
           contact_window_minutes — 429 Too Many Requests, JSON { ok: false }.

  Шаг C.4. Продумать, что X-Forwarded-For на shared-хостинге может
           передавать реальный IP — брать с осторожностью, только из доверенного
           прокси, иначе опирайтесь на REMOTE_ADDR.


--------------------------------------------------------------------------------
7. Этап D: правка фронтенда
--------------------------------------------------------------------------------

  Шаг D.1. В contacts/contacts.html в форму добавить СКРЫТОЕ поле honeypot
           (например name="companyWebsite" с tabindex="-1" и
           class="... visually-hidden" / aria-hidden="true" style="position:absolute;
           left:-10000px" — без display:none в некоторых сценариях боты
           не заполняют; оставьте важное: поле пустое у людей, заполнено у
           ботов). На сервере: если непусто — «тихий успех» без письма.

  Шаг D.2. Под form или рядом с кнопкой: контейнер для сообщения об успехе/ошибке
           (роль status, aria-live="polite"), чтобы убрать alert.

  Шаг D.3. В contacts-form.js:
           - e.preventDefault() оставить;
           - сформировать объект полей, включая honeypot и captchaToken
             (из API виджета smartCaptcha.getResponse после прохождения капчи);
           - fetch('api/submit.php', { method: 'POST', headers: { 'Content-Type':
             'application/json' }, body: JSON.stringify(...) });
           - путь: от страницы contacts/ относительный 'api/submit.php' корректен.
           - Кнопку submit по умолчанию disabled; включать после callback капчи;
             на token-expired снова disabled. После ответа сервера — reset виджета,
             снова нужна новая проверка (токен одноразовый).
           - Обработка ответа: 200 + ok, 4xx, 5xx; на время запроса — disabled.
           - при успехе — form.reset(), сброс капчи, обновить счётчик символов
             комментария.

  Шаг D.4. Content-Security-Policy на contacts.html: указать в meta-теге
           script-src и frame-src (и при необходимости connect-src) для
           https://smartcaptcha.cloud.yandex.ru; иначе виджет не загрузится.
           fetch на тот же origin (api/submit.php) — обычно 'self' в connect-src
           уже разрешён.


--------------------------------------------------------------------------------
8. Этап E: выкладка, права, проверка
--------------------------------------------------------------------------------

  Шаг E.1. Залить на хост (FTP/файловый менеджер панели Beget или иной) папки
           contacts/ и стили, вручную создать на сервере config.local.php.
           Состав: contacts/api/*.php, phpmailer/, data/, .htaccess, config.local
           (только на сервере), contacts-form.js, contacts.html.

  Шаг E.2. Права: contacts/api/data/ доступен PHP на запись; не открывайте
           листинг каталога (Options -Indexes в .htaccess/панель).

  Шаг E.3. Проверка вручную:
           (1) валидная заявка после капчи — письмо на всех получателей, Reply-To
               на клиента;
           (2) отправка без прохождения капчи — нельзя (кнопка неактивна) или
               ошибка на клиенте/сервере (captcha);
           (3) пустые обязательные — 4xx, понятное сообщение;
           (4) слишком длинные строки — отказ;
           (5) много успешных отправок с одного IP — 429;
           (6) заполненный honeypot — письмо не ушло, ответ «успех» (тихо для бота);
           (7) смена/удаление captchaToken в DevTools — 400, error captcha.

  Шаг E.4. Убедиться, что config.local.php и rate-limit-файлы не попали в
           публичный репозиторий.


--------------------------------------------------------------------------------
9. Yandex Smart Captcha: ключи, CSP, проверка
--------------------------------------------------------------------------------

  9.1. Где взять ключи
    В Yandex Cloud создайте сервис Smart Captcha (документация:
    yandex.cloud → Smart Captcha, быстрый старт / проверка ответа).
    Для одной капчи выдаётся:
      - Клиентский ключ (sitekey) — вставьте в contacts.html в атрибут
        data-sitekey у элемента #contactSmartcaptcha.
      - Серверный ключ (secret) — в contacts/api/config.local.php в поле
        smartcaptcha_server_key. В репозиторий не коммитить.

  9.2. Поведение на сайте
    Скрипт https://smartcaptcha.cloud.yandex.ru/captcha.js (defer) подключается
    в contacts.html после общих скриптов сайта, перед contacts-form.js.
    Виджет рендерится в #contactSmartcaptcha; кнопка «Отправить» disabled, пока
    не сработает callback. При истечении токена — снова disabled (событие
    token-expired).

  9.3. Сервер
    submit.php передаёт в validate secret, token (из тела JSON как captchaToken)
    и ip. Успех: JSON { "status": "ok" }. Иначе клиенту — error: "captcha".

  9.4. Content-Security-Policy (meta на contacts.html)
    Минимум: script-src … https://smartcaptcha.cloud.yandex.ru;
    frame-src … https://smartcaptcha.cloud.yandex.ru; при необходимости
    connect-src … (если внутренние запросы виджета к API блокируются — смотрите
    консоль браузера и расширьте connect-src по доменам из ошибки).

  9.5. Ошибка «ключ не может использоваться на домене» (file:// и localhost)
    • Не открывайте contacts.html двойным щелчком (адрес в браузере начинается с
      file:///) — у такой страницы нет домена, Yandex не сопоставит ключ.
    • Поднимите каталог проекта как сайт, например из корня репозитория:
        python -m http.server 8000
      (или npx serve, или встроенный сервер в PHP: php -S 127.0.0.1:8000)
      и откройте: http://127.0.0.1:8000/contacts/contacts.html
    • В Yandex Cloud → Smart Captcha → ваша капча → разрешённые сайты/хосты
      добавьте: 127.0.0.1, localhost и реальный продакшн-домен (и при
      необходимости поддомены), иначе виджет покажет ту же ошибку.
    См. также: yandex.cloud → раздел «Решение проблемы» по Smart Captcha
    (Key cannot be used on the domain).

  9.6. Секрет в JavaScript не помещать — только server key в PHP.


--------------------------------------------------------------------------------
  10. Типовые проблемы
--------------------------------------------------------------------------------

  Капча не отображается / «Не удалось загрузить Yandex Smart Captcha»:
    — Проверьте data-sitekey (не пустой, тот же проект, что server key);
    — CSP: script-src, frame-src для smartcaptcha.cloud.yandex.ru; консоль
      (F12) на предмет «Refused to load/execute…»;
    — Блокировщики рекламы / режимы приватности: временно отключить для теста.

  Ошибка на форме / в ответе API с кодом captcha:
    — Несовпадение пары client/server key, истёкший или повторно использованный
      токен (после отправки виджет сбрасывается — пройдите капчу снова);
    — smartcaptcha_server_key пустой или неверный в config.local.php;
    — с сервера нет исходящего HTTPS к smartcaptcha.cloud.yandex.ru (редко
      на сильно закрытом хостинге) — смотрите error_log PHP.

  «Сервер не настроен…» (server_config) при заполненной почте:
    — Добавьте в config.local.php непустой smartcaptcha_server_key
      (и проверьте остальные обязательные поля, см. шаг 0.7).

  Письма не приходят (включая Beget):
    — логи на хосте (в панели Beget часто есть журнал ошибок PHP) и/или
      временное логирование в submit.php (только на отладке);
    — неверный smtp_user: у Beget это полный e-mail, не логин без @;
    — несоответствие порта и smtp_secure: 465 + 'ssl' (SMTPS) или, если
      используете 2525/587 по KB Beget, обычно 'tls' (STARTTLS) — смотрите
      текущую таблицу в базе знаний Beget;
    — from_email и smtp_user — один и тот же ящик, созданный в панели;
    письмо в «Спам»: SPF/записи для домена в панели Beget и приёмнике;
    — лимит rate limit: код error «rate» (раздел 0.3) — смените сеть/подождите.

  Проблемы SMTP у Beget (см. также KB, разделы про почту / ошибки подключения):
    https://beget.com/ru/kb/how-to/mail/
    Часто помогает смена пары порт+шифр (465/ssl ↔ 2525/tls) по рекомендациям
    из статьи на момент настройки.

  403/404 на api/submit.php:
    — путь; на Beget/Apache .htaccess в contacts/api/ уже в репозитории;
      проверьте, что папка и имя submit.php в нижнем регистре, как в Linux;
      путь в браузере: …/contacts/api/submit.php при прямом тесте.

  CORS-ошибка в консоли:
    — чаще всего file:// или другой поддомен; тест с https://ваш-домен/… с сайта
      на Beget, тот же origin.

  «Пустой» $_POST:
    — форма шлёт JSON — в submit.php тело читается из php://input, не $_POST.

  (в т.ч. smartcaptcha_server_key) и проверьте тестовую заявку; затем — боевой
  список recipients.
