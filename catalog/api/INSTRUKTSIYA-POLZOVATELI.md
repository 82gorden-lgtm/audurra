# Инструкция: пользователи «Мой New Site» и правки каталога

Документ описывает, как устроены **вход без регистрации**, **добавление и смена редакторов** и **сохранение правок карточек товаров** на сайте New Site. Это соответствует сценарию: специальные пользователи задаются **только на сервере** (секреты не в клиентском JavaScript), окно входа открывается **только по клику** на ссылку **«Мой New Site»** в шапке сайта.

---

## 1. Как это работает в целом

1. Посетитель нажимает **«Мой New Site»** — открывается модальное окно с полями **логин** и **пароль** (отдельной формы регистрации нет).
2. Браузер отправляет данные на **`catalog/api/login.php`**. Проверка выполняется **только на сервере** (PHP).
3. Учётные записи хранятся в файле **`catalog/api/config.local.php`**: для каждого логина хранится **хеш пароля** (`password_hash`), при необходимости — ограниченный список зон редактирования (см. п. 3.3 ниже).
4. После успешного входа создаётся **сессия** (cookie). Учитываются права: на **странице товара** может отображаться **«Редактировать карточку»** (зона **`catalog`**), правки сохраняются в **`catalog/api/data/product-overrides.json`**. Аналогично отдельно настраиваются карта дилеров (**`dealers`**) и редактируемые тексты страниц (**`page_text`**).

Файлы API (PHP) должны выполняться на хостинге с поддержкой PHP (например, **Beget**). Статический деплой по FTP не отменяет необходимость **вручную** положить на сервер `config.local.php` и выставить права на каталог `data/`, если это ещё не сделано.

---

## 2. Файлы, с которыми вы работаете

| Файл | Назначение |
|------|------------|
| **`config.local.sample.php`** | Образец структуры конфигурации (можно коммитить в репозиторий). |
| **`config.local.php`** | Рабочий файл с пользователями — **на сервере**; в git обычно **не коммитится** (см. `.gitignore`). |
| **`data/product-overrides.json`** | Накопленные правки карточек по SKU; создаётся/обновляется PHP при сохранении из редактора. |
| **`login.php`**, **`logout.php`**, **`session.php`**, **`save-product.php`**, **`save-dealers.php`**, **`save-page-text-overrides.php`**, **`read-overrides.php`** | Логика входа, сессии и данных (правки возможны только в разрешённых для учётной записи зонах). |

## 3. Подготовка `config.local.php` на сервере

### 3.1. Если файла ещё нет

1. Скопируйте **`config.local.sample.php`** в **`config.local.php`** в каталоге **`catalog/api/`** на сервере (тот же уровень, что и `bootstrap.php`).
2. Откройте **`config.local.php`** в редакторе или в файловом менеджере хостинга.

### 3.2. Базовая структура (полный доступ ко всем зонам редактирования)

Файл должен возвращать массив с ключом **`users`** — ассоциативный массив **логин → хеш пароля**:

```php
<?php

declare(strict_types=1);

return [
    'users' => [
        'имя_логина' => '$2y$10$..............................................................',
    ],
];
```

- **Логин** — строка без пробелов в начале/конце (удобно: латиница и цифры).
- **Значение** — одна строка, которую вернула функция PHP **`password_hash`** (часто начинается с **`$2y$10$`**).

Пароль в этом файле **никогда не храните** в открытом виде.

### 3.3. Ограничение зон для отдельного пользователя

Если вместо строки-хеша указать **массив** с ключом **`password`** и списком **`can`** (или синоним **`capabilities`**), учётная запись сможет сохранять изменения **только** в перечисленных зонах:

| Ключ | Где на сайте |
|------|----------------|
| **`catalog`** | Карточки товаров в каталоге (`save-product.php`). |
| **`dealers`** | Представители на карте «Где купить» (`save-dealers.php`). |
| **`page_text`** | Редактируемые текстовые блоки на страницах (`save-page-text-overrides.php`). |

Пример: только карта дилеров:

```php
'users' => [
    'karta' => [
        'password' => '$2y$10$..............................................................',
        'can' => ['dealers'],
    ],
],
```

Если у записи в **массиве** нет ключей `can` / `capabilities`, поведение как у старого формата — **все три зоны**. Пустой массив **`'can' => []`** означает, что после входа кнопок редактирования не будет (сохранения на сервере тоже будут отклонены).

Права перечитываются с **каждым запросом** к API из `config.local.php`, поэтому смена списка `can` применяется без повторного входа.

## 4. Пошаговое добавление нового пользователя

### Шаг 1. Сгенерируйте хеш пароля

**Вариант A — PHP на вашем компьютере** (если установлен PHP):

```bash
php -r "echo password_hash('Ваш_надежный_пароль', PASSWORD_DEFAULT), PHP_EOL;"
```

Скопируйте **целиком** одну строку вывода (без пробелов в начале/конце).

**Вариант B — временный скрипт только на сервере** (если локально нет PHP):

1. Создайте **временный** файл (например, вне публичной зоны или с уникальным именем), который выводит только `password_hash` для выбранного пароля.
2. Откройте его в браузере **один раз**, скопируйте результат.
3. **Удалите** временный файл с сервера, чтобы им нельзя было воспользоваться повторно.

### Шаг 2. Добавьте строку в `config.local.php`

Откройте **`config.local.php`** и в массив **`users`** добавьте пару логин → хеш:

```php
'users' => [
    'redaktor1' => '$2y$10$xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx',
    'novyy_login' => '$2y$10$yyyyyyyyyyyyyyyyyyyyyyyyyyyyyyyyyyyyyyyyyyyyyyyyyyyyyyyyyyyy',
],
```

Сохраните файл в кодировке **UTF-8**.

### Шаг 3. Загрузите файл на сервер (если правили локально)

Убедитесь, что актуальный **`config.local.php`** лежит в **`catalog/api/`** относительно корня сайта (как в URL: `https://ваш-домен.ru/catalog/api/...`).

### Шаг 4. Проверьте вход

1. Откройте сайт по **HTTPS**.
2. Нажмите **«Мой New Site»** (модальное окно открывается **только** по этому действию).
3. Введите **логин** и **пароль**, для которого строили хеш.

При пустом или отсутствующем **`config.local.php`** возможна ошибка в духе «вход не настроен на сервере». При неверной паре логин/пароль — сообщение о неверных учётных данных.

---

## 5. Несколько пользователей

Добавляйте несколько строк в **`users`** с **разными** логинами. У каждого свой хеш (даже если пароли совпадают — хеши могут отличаться, это нормально).

Удаление пользователя: удалите соответствующую строку из **`users`**.

Смена пароля: сгенерируйте новый **`password_hash`** для нового пароля и **замените** только строку хеша у нужного логина.

---

## 6. Права на каталог `data/` и сохранение карточек

Чтобы кнопка **«Сохранить»** в редакторе карточки товара работала, веб-сервер должен иметь право **записывать** в каталог:

**`catalog/api/data/`**

Там создаётся или обновляется **`product-overrides.json`**. Если запись запрещена правами, сохранение завершится ошибкой (сообщение о неудачной записи на сервере).

Деплой по FTP **может не заливать** `product-overrides.json`, если файл только на сервере — это ожидаемо: правки живут на хостинге. Каталог **`data/`** в репозитории обычно содержит только **`.gitkeep`**; сам JSON при необходимости создаёт PHP.

---

## 7. Безопасность (кратко)

- Не публикуйте **`config.local.php`** в открытом репозитории и не отправляйте его в незащищённую переписку.
- Используйте **длинные уникальные пароли** для редакторов.
- Сайт для входа лучше открывать по **HTTPS**, чтобы сессионные cookie не передавались открытым текстом по сети.
- После генерации хеша через временный скрипт на сервере **удаляйте** этот скрипт.

---

## 8. Связь с деплоем (Beget)

В комментариях к workflow деплоя указано: после выкладки файлов по FTP **один раз** настроить на сервере **`config.local.php`** и права на **`catalog/api/data/`**. Эта инструкция как раз описывает первую часть (пользователи); вторую — в разделе 6.

---

## 9. Влитие `product-overrides.json` в канон каталога (Git)

Чтобы правки с сервера стали частью репозитория, скачайте актуальный **`product-overrides.json`** в **`catalog/api/data/`** (или укажите путь флагом) и выполните из **корня репозитория**:

`python catalog/merge-overrides-into-catalog.py`

Подробная пошаговая инструкция в текстовом файле: **`catalog/INSTRUKTSIYA-OBNOVLENIE-KATALOGA-IZ-OVERRIDES.txt`**.

Полезные флаги: **`--dry-run`** (только статистика, без записи), **`--no-backup`** (не создавать `catalog-products-data.js.bak`). Логика слияния совпадает с **`site-editor.js`**. После прогона проверьте **`git diff`** на **`catalog/catalog-products-data.js`**, закоммитьте при необходимости и при желании упростите overrides на хостинге для уже влитых SKU.

## 10. Устранение неполадок

| Симптом | Возможная причина |
|--------|-------------------|
| «Вход не настроен» / `not_configured` | Нет **`config.local.php`**, пустой **`users`**, или PHP не читает файл (путь, права). |
| «Неверный логин или пароль» | Опечатка, неверный хеш, обрезана строка `$2y$...`, неверная кодировка файла. |
| Вход есть, сохранение не работает | Нет прав записи в **`catalog/api/data/`** или ошибка PHP при записи JSON. |
| Модалка не открывается | Должен быть клик по **«Мой New Site»**; при проблемах со стилями проверьте, что у скрытого оверлея задано **`display: none`** для атрибута **`hidden`**. |

---

*Текст отражает поведение фронтенда (`site-editor.js`, модалка только по клику, без подписи «войдите, чтобы редактировать…»), скрипт влития overrides (`catalog/merge-overrides-into-catalog.py`) и бэкенда в **`catalog/api/`** по состоянию проекта New Site.*
