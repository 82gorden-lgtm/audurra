# «Мой New Site»: учётные записи и три зоны редактирования

Единое руководство по добавлению редакторов, правам доступа и цепочке **сайт → файлы на сервере → Git**. Три зоны (каталог, представители на карте, тексты страниц) описаны **одинаково по структуре**, в том числе путь **переноса правок в репозиторий** — для дилеров он оформлен так же, как для каталога и текстовых блоков.

На хостинге должны выполняться скрипты PHP в `catalog/api/`. Регистрации на сайте нет.

---

## 1. Вход и где хранятся секреты

1. Пользователь открывает ссылку **«Мой New Site»** в шапке сайта и вводит логин и пароль.
2. Проверка выполняется в **`catalog/api/login.php`**. Пароль в конфиге хранится только как **хеш** (`password_hash` в PHP).
3. Успешный вход создаёт сессию в браузере. Для каждой учётки задан набор **зон** (`capabilities`). Без соответствующей зоны интерфейс редактирования для неё скрыт, а прямой вызов API сохранения вернёт код **`forbidden_capability`**.
4. Права читаются из **`catalog/api/config.local.php`** при каждом запросе (смена списка зон действует без повторного входа).

Рекомендуется использовать **HTTPS**.

---

## 2. Общее для всех зон: каталог данных на сервере

Чтобы сохранение с сайта работало, у процесса PHP должна быть **запись** в каталог:

**`catalog/api/data/`**

Если каталог недоступен для записи, все три типа сохранения закончатся ошибкой.

---

## 3. Конфигурация пользователей (`config.local.php`)

### Где файл

Файл **`catalog/api/config.local.php`** располагается рядом с `bootstrap.php`. Образец: **`catalog/api/config.local.sample.php`**. Рабочий файл на сервере обычно **не добавляют в Git**.

Базовая форма возвращаемого массива:

```php
<?php

declare(strict_types=1);

return [
    'users' => [
        // записи см. ниже
    ],
];
```

Кодировка **UTF-8**.

### Варианты записи пользователя

**Полный доступ** (все три зоны) — классическая форма «логин → строка-хеш»:

```php
'users' => [
    'operator' => '$2y$10$..............................................................',
],
```

**Выборочные зоны** — массив с **`password`** (хеш) и **`can`** или **`capabilities`** (одинаковый смысл):

```php
'users' => [
    'karta' => [
        'password' => '$2y$10$..............................................................',
        'can' => ['dealers'],
    ],
],
```

Строки, которые можно указывать в массиве прав:

| Значение | Зона на сайте |
|----------|----------------|
| **`catalog`** | Карточки товаров в каталоге |
| **`dealers`** | Представители на странице «Где купить» |
| **`page_text`** | Текстовые блоки с ключом `data-cromax-text-key` |

Комбинируйте нужные элементы или перечислите все три. Пример только каталога и карты:

```php
'two_zones' => [
    'password' => '$2y$10$...',
    'capabilities' => ['catalog', 'dealers'],
],
```

**Массив без `can` / `capabilities`** — снова **полный доступ** ко всем зонам (как строка-хеш):

```php
'all_implicit' => [
    'password' => '$2y$10$...',
],
```

**Явно без прав на редактирование** (вход возможен, кнопок сохранения нет):

```php
'no_edit' => [
    'password' => '$2y$10$...',
    'can' => [],
],
```

### Хеш пароля

Выполните на машине с PHP:

```bash
php -r "echo password_hash('Ваш_пароль', PASSWORD_DEFAULT), PHP_EOL;"
```

Скопируйте всю строку результата. Пароль в открытом виде в `config.local.php` **не записывают**.

### Смена и удаление

Смена пароля — новый хеш для того же логина. Удаление пользователя — удалить его блок из **`users`**.

## 4. Зона «каталог» (`catalog`)

### Редактирование на сайте

Откройте **страницу товара** в каталоге. После входа с правом **`catalog`** появится кнопка **«Редактировать карточку»**. Сохранение отправляет данные на **`catalog/api/save-product.php`** и после успеха обновляет страницу.

### Файл на сервере

Правки накапливаются в **`catalog/api/data/product-overrides.json`** (по SKU). Этот файл в типичном сценарии **не коммитят**: он указан в `.gitignore`; на сервере он живёт только на хостинге.

### Перенос в репозиторий («влить» в канон каталога)

Чтобы содержание карточек стало частью **`catalog/catalog-products-data.js`** в Git:

1. Скачайте с сервера актуальный **`product-overrides.json`** (положите при необходимости в **`catalog/api/data/`** локально или укажите путь).
2. Из корня репозитория выполните:

   `python catalog/merge-overrides-into-catalog.py`  
   (пробный просмотр: добавьте **`--dry-run`**).

Подробности, флаги и проверки: **`catalog/INSTRUKTSIYA-OBNOVLENIE-KATALOGA-IZ-OVERRIDES.txt`**.

## 5. Зона «представители / дилеры» (`dealers`)

### Редактирование на сайте

Откройте страницу **`who-we-are/where-to-buy.html`** (или её адрес после публикации). При входе с правом **`dealers`** доступна панель редактирования представителей по регионам. Сохранение идёт на **`catalog/api/save-dealers.php`**.

Чтение данных для страницы выполняется через **`catalog/api/read-dealers.php`** (без авторизации посетителя).

### Файл на сервере и в Git

Единый документ в формате JSON лежит в **`catalog/api/data/dealers-ru.json`**. В отличие от `product-overrides.json`, он **может версионироваться в репозитории** как эталон для деплоя; на хостинг при выкладке подставляется та же версия файла из репозитория (или обновление идёт с сервера после правок редакторов).

### Перенос в репозиторий (канон того же файла)

Поток тот же по смыслу, что у каталога и текстов: с сервера забирается JSON, приводится к одному виду с PHP и записывается в путь в клоне.

1. Скачайте **`dealers-ru.json`** с хостинга из **`catalog/api/data/`** (или подмените им локальный файл в том же пути перед шагом 2).
2. Из корня репозитория:

   `python catalog/merge-dealers-json-into-repo.py`  
   (пробный просмотр: **`--dry-run`**; свой путь к скачанному файлу: **`--source`**).

Полное описание, резервная копия `.bak`, отличие от статического пайплайна каталога/текстов: **`catalog/INSTRUKTSIYA-VLITYe-DILERY-JSON.txt`**.

**Сводка-параллель:**

| Элемент | Каталог | Дилеры | Тексты страниц |
|--------|---------|--------|----------------|
| Python-скрипт | `catalog/merge-overrides-into-catalog.py` | `catalog/merge-dealers-json-into-repo.py` | `catalog/merge-page-text-overrides-into-html.py` |
| Инструкция в репо | `catalog/INSTRUKTSIYA-OBNOVLENIE-KATALOGA-IZ-OVERRIDES.txt` | `catalog/INSTRUKTSIYA-VLITYe-DILERY-JSON.txt` | `catalog/INSTRUKTSIYA-VLITYe-TEXT-BLOKOV-HTML.txt` |

## 6. Зона «тексты страниц» (`page_text`)

### Редактирование на сайте

Разметкой отмечаются элементы с атрибутом **`data-cromax-text-key`** и уникальным ключом. Редактор подставляет **простой текст**; фрагменты со сложной вложенной HTML-разметкой внутри одного узла для этого режима не подходят.

Страницы **каталога** и **«Где купить»** в проекте **исключены** из общего текстового редактора (у них свои интерфейсы из разделов 4 и 5).

После входа с правом **`page_text`** на остальных страницах с ключами работает панель правки (как реализовано в **`site-editor.js`**). Сохранение — **`catalog/api/save-page-text-overrides.php`**.

### Файл на сервере

**`catalog/api/data/page-text-overrides.json`**.

### Перенос в репозиторий

Скрипт переносит блоки в **статические HTML** репозитория:

`python catalog/merge-page-text-overrides-into-html.py` (с **`--dry-run`** при необходимости).

Инструкция: **`catalog/INSTRUKTSIYA-VLITYe-TEXT-BLOKOV-HTML.txt`**.

## 7. Сводная таблица API и файлов

| Зона | Право в `can` | Сохранение (POST) | Файл данных (сервер) |
|------|----------------|-------------------|------------------------|
| Каталог | `catalog` | `catalog/api/save-product.php` | `catalog/api/data/product-overrides.json` |
| Представители | `dealers` | `catalog/api/save-dealers.php` | `catalog/api/data/dealers-ru.json` |
| Тексты | `page_text` | `catalog/api/save-page-text-overrides.php` | `catalog/api/data/page-text-overrides.json` |

---

## 8. Типичные проблемы

| Симптом | Проверка |
|---------|----------|
| Не пускает в «Мой New Site» | Есть ли `config.local.php`, заполнен ли `users`, верен ли хеш |
| Нет нужной кнопки / панели | Есть ли зона в `can`; открыта ли правильная страница |
| Ошибка **`forbidden_capability`** | Добавьте зону в `can` или используйте полный доступ |
| Не записывается на сервер | Права на запись в **`catalog/api/data/`** |
| После вливания в Git сайт «откатился» | Убедитесь, что деплой залил обновлённые канонические файлы (JS / HTML / `dealers-ru.json` по вашему процессу) |

---

*Актуальность: зоны `catalog`, `dealers`, `page_text`; скрипты и пути в каталоге `catalog/`.*
